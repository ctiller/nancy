# 0051. Stateless Dreamer Hydration

## Title
Dreamer Caching & Stateless Hydration

## Context
The administrative "Dreamer" node is responsible for monitoring all `Event` payloads securely and executing complex asynchronous task ranking logic (specifically executing `LlmClient` prompts asynchronously). Previously, the Dreamer explicitly iterated and re-evaluated the entire event history of the project log every tick upon restarting, leading to heavy redundant LLM API toll, massive execution delays on hydration, and unstable memory overhead mapping large DAG structures.

## Decision
We extended the `TaskViewEvaluator` state-matrix natively. Before evaluating events, the module queries the branch log parsing existing `TaskEvaluation` payload checkpoints structurally generated by past executions. 
Instead of discarding memory gracefully, the Dreamer caches these explicit node tracking results in `evaluated_event_ids`. At hydration, it drops early, only committing LLM network requests to fresh log operations cleanly.

## Consequences
1. Re-running the `dreamer` pipeline safely resumes exactly where the execution halted natively without any token loss.
2. Changes to evaluation constraints inherently require clearing or manually editing registry node outputs to force LLM re-run evaluations mechanically.
