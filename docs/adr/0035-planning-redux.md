# 0035: Planning Redux - Native Grinder Multi-Agent Synthesis

## Title
Task Pipeline Redux: In-Grinder Ideation, Synthesis, and Artifact CI

## Context
Our previous planning system (`TaskAction::Plan -> TaskAction::ReviewPlan`) lacked divergent thinking and produced rigid, single-threaded plans. The review ecosystem was explicitly reliant on generating separate Git `nancy/plans/<request_id>` branches for design ecosystems. It also supported paused states via `review_session_file`, necessitating long-running conversation persistence across event ledger boundaries which caused orchestration gridlocks.

To boost the quality of architectural planning and implementation without exposing the root `Coordinator` to massive LLM DAG scaling, we are shifting the structural approach away from formal Git tracking for logical tasks and instead mapping multi-agent orchestration safely into an isolated local agent footprint.

## Decision
We are comprehensively refactoring the Plan generation and Review protocols to embrace stateless ledgers and rehydratable workflows:

### 1. Unified Grinder Planning Function & No-Branch Strategy
The `Coordinator` dispatches a `TaskAction::Plan` task to the Grinder. Planning tasks **no longer create specialized Git branches**. A plan is purely conceptual and its output doesn't require an isolated version control footprint, though the active context maintains access to target the implementation codebase.

Within the Grinder footprint, the single task maps to a multi-agent workflow:
- **Moderator Setup**: Provide the initial task to a Moderator LLM and ask it to define a list of distinct contributor roles/perspectives relevant to the task.
- **Divergent Ideation**: Launch $N$ `thinking_llm` instances concurrently. Each contributor LLM receives the task and produces an independent drafted component.
- **Synthesis**: The Moderator consumes all drafts and synthesizes the authoritative plan.

### 2. Schema Redesign and Artifact Persistence
`TaskPayload` explicitly drops legacy paused-state mechanics and replaces them with a native file pointer mapping.
- `review_session_file` is fully deprecated.
- `plan: Option<String>` is introduced.
Output generated by the synthesized planning run places an `.md` architecture summary strictly into `plans/:id.md` in the `.nancy/agents/` local directory. The structured task breakdowns generated alongside the plan are intrinsically parsed and loaded into the SQLite Event Ledger as generic `Task` instances bounded via `BlockedBy` edges.

### 3. Plan Continuous Integration (CI)
A deterministic validation process executes prior to saving plan completion state.
- Ensures `plans/:id.md` physically exists.
- Validates the synthesized JSON task breakdowns against strict schema structures.
Programmatic validation failures halt synthesis completion and bounce back iteratively to the Moderator for immediate format correction without polluting the Review round.

### 4. Stateless, Non-Pausable Reviews & Feedback Resynthesis
`TaskAction::ReviewPlan` and `TaskAction::ReviewImplementation` are universally stateless and not intrinsically pausable. The Veto classification system holds.

If a panel triggers a `ChangesRequired` vote or a `Veto` rejection action:
- Feedback metadata logs are routed contextually to the original planning constraints.
- The Grinder intrinsically **rehydrates the exact original planning context** (the existing Moderator and $N$ active `thinking_llms`).
- The bounded session resynthesizes and outputs a patched functional iteration resolving the reviewer dissent, looping until consensus is reached.

## Consequences
- Physical footprint scale drops astronomically for complex plans by eliminating unnecessary plan branch Git commits.
- High structural task capability manages scaling exclusively through LLM sub-roles isolating orchestrator complexity boundaries tightly off the SQLite framework. 
- Deterministic schema-checks ensure the Coordinator never crashes mapping raw invalid LLM instruction hallucination formats.
